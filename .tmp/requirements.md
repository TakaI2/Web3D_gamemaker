# 要件定義書 - FPS ステージエディタ

## 1. 目的

FPSゲーム（`FpsViewport.svelte` ベース）のステージを視覚的に構築するための 3D エディタを提供する。
現在はハードコードされた `collision-world.glb` をマップとして使用しているが、エディタで作成したステージを GLB エクスポートし、FPS ゲームに読み込ませることで、ステージ制作を容易にする。

RPG-Game の 2D マップチップエディタに相当する機能を、Three.js ベースの 3D 空間で実現する。

---

## 2. 機能要件

### 2.1 必須機能

#### 2.1.1 3D ビューポート
- [ ] Three.js によるリアルタイム 3D レンダリング
- [ ] オービットカメラ（マウスドラッグで回転、ホイールでズーム、右ドラッグでパン）
- [ ] XZ 平面グリッドの表示（視覚的な配置補助）
- [ ] ビューポートのリサイズ対応

#### 2.1.2 基本図形の配置
- [ ] 配置可能な図形: Box（直方体）、Sphere（球）、Cylinder（円柱）、Cone（円錐）
- [ ] グリッドスナップ: 設定した間隔（0.5 / 1 / 2 / 4 単位）で座標を丸めて配置
- [ ] 配置プレビュー: マウスオーバー時に半透明ゴーストで配置位置を表示
- [ ] クリックで確定配置
- [ ] 配置後もサイズ（X/Y/Z スケール）を個別に設定可能

#### 2.1.3 オブジェクト選択・操作
- [ ] クリックでオブジェクト選択（ハイライト表示）
- [ ] 選択オブジェクトの位置（X/Y/Z）をプロパティパネルで数値入力
- [ ] 選択オブジェクトの回転（X/Y/Z 度）をプロパティパネルで数値入力
- [ ] 選択オブジェクトの削除（Delete キー または ボタン）
- [ ] 配置モード ↔ 選択モードの切り替え

#### 2.1.4 マテリアル・テクスチャ適用
- [ ] カラー（color）の設定: カラーピッカー
- [ ] Roughness / Metalness のスライダー調整
- [ ] ローカル画像ファイル（PNG/JPG）のテクスチャとしてアップロード・適用
- [ ] マテリアル設定はオブジェクト単位で保持

#### 2.1.5 シーン保存・読み込み
- [ ] シーン全体を JSON 形式でブラウザにダウンロード保存
- [ ] JSON ファイルをアップロードしてシーンを復元（再編集可能）
- [ ] JSON には全オブジェクトの図形種別・位置・回転・スケール・マテリアル情報を含む

#### 2.1.6 GLB エクスポート / シーン読み込み（FPS 向け）
- [ ] Three.js `GLTFExporter` を使って GLB ファイルをエクスポート
- [ ] エクスポートした GLB を FPS ゲームの `collision-world.glb` として使用できる形式にする（Mesh + Material を含む）
- [ ] テクスチャ画像を GLB に埋め込む

#### 2.1.7 FPS プレビューモード（必須）
- [ ] エディタ画面から「FPS でテスト」ボタンを押すと、現在のシーンを一時 GLB に変換して FPS ゲームに渡す
- [ ] FPS ゲーム終了後（「エディタへ戻る」ボタン）、エディタの状態を維持したまま復帰できる
- [ ] 既存アプリのトップページから「ステージエディタ」ボタンでアクセスできる

### 2.2 オプション機能（将来対応）

- [ ] Undo / Redo（Ctrl+Z / Ctrl+Y）
- [ ] 複数オブジェクトの一括選択・操作
- [ ] コピー＆ペースト
- [x] FPS プレビューモード → **必須機能に昇格**（下記 2.1.7 参照）
- [ ] オブジェクトのグループ化・階層管理
- [ ] スポーン地点（プレイヤー開始位置）の配置
- [ ] 天井・壁の自動生成補助（部屋ウィザード）
- [ ] GLB/GLTF モデルのインポートと配置

## 3. 非機能要件

### 3.1 パフォーマンス
- 100 オブジェクト程度のシーンで 60fps 維持（Three.js の InstancedMesh は MVP 後に検討）
- テクスチャ適用後もリアルタイムプレビューが維持されること

### 3.2 セキュリティ
- テクスチャ画像はブラウザ内のみで処理（外部サーバーへのアップロードなし）
- JSON/GLB はすべてローカルダウンロード

### 3.3 保守性
- 既存の Svelte + TypeScript + Three.js アーキテクチャに従う
- `any` / `unknown` 型を使用しない
- `class` を使わずファクトリ関数パターンで実装（既存コードの規約に合わせる）
- ロジックを Svelte コンポーネントから分離（`src/stage-editor/` ディレクトリを新設）

### 3.4 互換性
- エクスポートした GLB が `FpsWorld.ts` の `GLTFLoader` + `Octree.fromGraphNode()` で正常に動作すること
- 既存の `vite.config.ts` のビルド設定・マルチページ構成に統合

---

## 4. 制約事項

### 4.1 技術的制約
- Three.js（既存バージョン）を使用
- Svelte（既存バージョン）でUI構築
- TypeScript strict モード
- GLB エクスポートは `three/addons/exporters/GLTFExporter.js` を使用
- ブラウザ完結（Node.js サーバーサイド処理なし）

### 4.2 スコープ制約
- アニメーション・物理シミュレーションはエディタスコープ外
- マルチユーザーリアルタイム編集はスコープ外

---

## 5. 成功基準

### 5.1 完了の定義
- [ ] Box を 10 個グリッドスナップで配置できる
- [ ] 各 Box に異なる色・テクスチャを適用できる
- [ ] シーンを JSON に保存 → 再度読み込んで完全復元できる
- [ ] GLB エクスポート → FPS ゲームで `collision-world.glb` として読み込み、Octree 衝突が機能する
- [ ] エディタが既存アプリ（VRMエディタ等）と同一 Vite プロジェクトからアクセスできる

### 5.2 受け入れテスト
- ステージを作成してFPSゲームで実際に歩き回れること
- 保存→読み込みサイクルでシーンが完全に復元されること

---

## 6. 想定されるリスク

| リスク | 影響 | 対策 |
|---|---|---|
| GLTFExporter の Octree 互換性 | 高 | エクスポート直後に FPS でテスト検証 |
| テクスチャ埋め込みで GLB サイズ肥大化 | 中 | テクスチャ解像度の上限設定（例: 1024px） |
| ブラウザの Blob URL メモリリーク | 中 | テクスチャ破棄時に `URL.revokeObjectURL` を呼ぶ |
| Raycaster と Orbit Controls の競合 | 低 | イベント制御の優先順位を明確化 |

---

## 7. 今後の検討事項（設計フェーズで詳細化）

- エントリーポイントの追加方法（`stage-editor-main.ts` + `vite.config.ts` の `input` 追加 vs. 既存アプリへの統合）
- グリッドスナップの実装方法（Raycaster → XZ 平面との交点 → `Math.round(v / snap) * snap`）
- オービットコントロールの実装（`OrbitController.ts` の流用 vs. three/addons の `OrbitControls` 使用）
- マテリアル設定の状態管理（Svelte store vs. コンポーネントローカル state）
- シーン JSON のスキーマ定義（型安全な読み書き）
- オブジェクトリストの UI レイアウト（左サイドバー vs. 下部パネル）
