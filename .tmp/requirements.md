# 要件定義書 - VRM対応3Dキャラクターエディタ（Web/WebXR）

## 1. 目的

Web ブラウザ上で動作する VRM キャラクターエディタを構築する。
VRM モデルのビューア・アニメーション再生・リップシンクプレビューを提供し、
Quest 3 を対象とした WebXR（VR/AR 切り替え）に対応する。
将来的にゲームモードと共通の基盤となる技術スタックを確立することも目的とする。

---

## 2. 機能要件

### 2.1 必須機能

#### モデル読み込み
- [ ] VRM ファイル（VRM 0.x / VRM 1.0 両対応）をドラッグ&ドロップまたはファイル選択ダイアログで読み込む
- [ ] 読み込み中はローディングインジケーターを表示する
- [ ] 読み込み失敗時（非対応フォーマット・破損ファイル等）はエラーメッセージを表示する
- [ ] 複数 VRM の切り替え（再読み込みで既存モデルを置換）

#### 3D ビューポート
- [ ] オービットコントロール：マウスドラッグで回転、スクロールでズーム、右クリックドラッグでパン
- [ ] タッチ操作対応（ピンチズーム・スワイプ回転）
- [ ] グリッドフロア表示（オン/オフ切り替え可能）
- [ ] 環境照明（DirectionalLight + AmbientLight の基本セット）
- [ ] モデルを画面中央にフィットさせるリセットボタン

#### アニメーション
- [ ] VRMA ファイル（複数）をドラッグ&ドロップまたはファイル選択で追加読み込み
- [ ] 読み込んだアニメーション一覧をリスト UI で表示（ファイル名ベース）
- [ ] リストからアニメーションを選択して再生・停止・ループ切り替え
- [ ] アニメーション再生中のタイムライン表示（現在時刻 / 総時間）とシーク操作
- [ ] Tポーズ（デフォルトポーズ）へのリセット機能
- [ ] アニメーション再生速度の調整（0.25x / 0.5x / 1.0x / 2.0x）

#### Spring Bone（揺れもの）
- [ ] VRM 内蔵の Spring Bone を自動有効化（髪・スカート・胸など）
- [ ] Spring Bone のオン/オフ切り替え UI
- [ ] Spring Bone の剛性・減衰パラメータのリアルタイム調整スライダー（デバッグ用）

#### リップシンク（テキストベース）
- [ ] テキスト入力フィールドにセリフを入力すると口パクアニメーションが再生される
- [ ] 日本語テキストを仮名分解し、母音（あいうえお）を VRM Viseme（aa/ih/ou/ee/oh）にマッピング
- [ ] タイプライター表示と同期して口パクが進行する（1 文字表示 → 対応 Viseme 適用）
- [ ] 発話速度（文字/秒）の調整スライダー
- [ ] リップシンク停止時は口を自然な閉じた状態（neutral）に戻す

#### WebXR 対応
- [ ] VR モード開始ボタン（`immersive-vr`セッション）
- [ ] AR モード開始ボタン（`immersive-ar`セッション、Quest 3 カラーパススルー）
- [ ] 非対応ブラウザでは各ボタンをグレーアウトしてツールチップで説明
- [ ] XR セッション中もオービットコントロールに相当する視点操作（コントローラーによる移動）
- [ ] XR セッション終了ボタン（ヘッドセット内 UI）
- [ ] VR ↔ AR 切り替えはセッション終了→再開始で対応（UI 上で明示）

### 2.2 オプション機能（低優先度）

- [ ] 布シミュレーション（ammo.js Softbody）：WebXR 非使用時のみ有効化
- [ ] HDRI 環境マップの読み込みと切り替え
- [ ] VRM メタデータ表示パネル（作者名・利用条件等）
- [ ] スクリーンショット保存（PNG エクスポート）
- [ ] カメラプリセット（正面・背面・側面・俯瞰）
- [ ] 複数ライトの追加・位置調整
- [ ] VRM BlendShape / Expression の手動スライダー操作
- [ ] アニメーションのブレンド（クロスフェード）

## 3. 非機能要件

### 3.1 パフォーマンス

- WebXR（Quest 3）で 72fps 以上を維持すること（Quest 3 の標準リフレッシュレート）
- VRM 読み込み完了まで 5 秒以内（一般的な VRM ファイル ~10MB 想定）
- Spring Bone 演算を含むフレーム処理が 13ms（72fps の 1 フレーム予算）以内に収まること
- メモリ使用量：GPU メモリ 512MB 以内を目標とする

### 3.2 セキュリティ

- VRM / VRMA ファイルはすべてローカル読み込みのみ（外部 URL からの取得は行わない）
- ユーザー入力テキストは DOM への直接挿入を避け XSS を防止する

### 3.3 保守性

- TypeScript を使用し、`any` / `unknown` 型の使用を禁止する
- コンポーネント単位で分割し、エディタ機能をゲームモードから独立して再利用できる構造とする
- Vite によるモジュールバンドルを使用し、開発時 HMR を有効にする

### 3.4 互換性

- **ターゲットブラウザ**: Quest 3 内蔵ブラウザ（Meta Browser）/ Chrome 最新版 / Firefox 最新版
- **VRM**: VRM 0.x および VRM 1.0 の両仕様に対応（@pixiv/three-vrm 使用）
- **アニメーション形式**: .vrma（VRM Animation 形式）のみ。Blender アドオンで事前変換を前提とする
- **入力デバイス**: マウス・キーボード・タッチスクリーン・Quest 3 Touch Plus コントローラー

## 4. 制約事項

### 4.1 技術的制約

| 項目 | 制約内容 |
|---|---|
| レンダリング | Three.js（WebGL 2.0 ベース） |
| VRM | @pixiv/three-vrm / @pixiv/three-vrm-animation |
| 物理（揺れもの） | @pixiv/three-vrm 内蔵 Spring Bone のみ（エディタ段階） |
| 物理（布） | 優先度低・後回し |
| ゲーム物理 | エディタには不要（ゲームフェーズで Rapier.js 導入） |
| ビルドツール | Vite + TypeScript |
| パッケージ管理 | npm または pnpm |
| 言語 | TypeScript（class は Error 継承のみ許可） |
| WebXR | VR↔AR はセッション再起動が必要（WebXR 仕様上の制約） |
| アニメーション形式 | .vrma のみ対応。FBX/BVH の直接読み込みはスコープ外 |

### 4.2 ビジネス制約

- GitHub リポジトリ `TakaI2/Web3D_gamemaker` で管理
- OSS ライブラリのライセンスは MIT または同等のものを使用すること

## 5. 成功基準

### 5.1 完了の定義

- [ ] VRM ファイルをブラウザ上で読み込み、3D モデルが正常に表示される
- [ ] マウス操作でモデルを任意の角度から閲覧できる
- [ ] VRMA アニメーションを読み込み、リストから選択して再生できる
- [ ] Spring Bone による揺れものが物理的に動作する
- [ ] テキスト入力に連動した口パクアニメーションが動作する
- [ ] Quest 3 ブラウザで VR モード・AR モードへの切り替えが動作する
- [ ] 72fps（Quest 3）でスムーズに動作する

### 5.2 受け入れテスト

- 任意の VRM ファイルを読み込んだとき、モデルが崩れず正常に表示されること
- アニメーション切り替え時に不自然なポーズジャンプが発生しないこと
- テキスト「こんにちは」を入力すると、5 文字分の口パクが順番に再生されること
- Quest 3 で WebXR ボタンを押すと VR / AR 両モードに入れること

## 6. 想定されるリスク

| リスク | 影響度 | 対策 |
|---|---|---|
| VRM 0.x と 1.0 の仕様差異 | 中 | @pixiv/three-vrm の抽象 API を使用してバージョン差を吸収 |
| Quest 3 ブラウザの WebXR 実装差異 | 高 | Meta Browser でのテストを早期に実施。Feature Detection で非対応機能をグレーアウト |
| Spring Bone 演算の高負荷化 | 中 | 高ポリゴン VRM での FPS 計測を早期に行い、ボーン数の上限を設定 |
| VRMA 形式への変換失敗 | 中 | Blender アドオンの動作検証を実装と並行して行う |
| リップシンクの仮名変換精度 | 低 | 外来語・記号は neutral（閉口）にフォールバックして対応 |
| ammo.js（布シミュ）の WebXR 負荷 | 低 | 優先度低のため、後フェーズで評価 |

## 7. 今後の検討事項（設計フェーズで詳細化）

- UI フレームワーク選定（純粋な HTML/CSS か、軽量 UI ライブラリか）
- VRM / VRMA ファイルのステート管理設計（複数ファイルの保持方法）
- WebXR セッション中の UI 配置（ヘッドセット内に浮かぶパネルの設計）
- Three.js の WebXRManager と OrbitControls の共存方法
- ゲームモードへのコード共有境界の設計（エディタとゲームで分離するべき責務）
- リップシンクのテキスト処理：ローマ字・英語テキストへの対応範囲
